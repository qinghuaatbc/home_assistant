## 1. 项目概述

本项目是一个完整的 Home Assistant 智能家居平台的 NestJS/TypeScript 重新实现。它使用 **NestJS** 作为后端框架、**TypeORM + better-sqlite3** 作为数据库、**React 18** 作为前端、**Three.js** 作为 3D 楼层平面渲染引擎、**Socket.IO** 作为实时通信协议。

### 核心架构

```
┌──────────────┐     HTTP/REST      ┌──────────────────────┐
│   React 前端  │ ◄──────────────►  │  NestJS 后端 (8123)   │
│  (Vite + TS)  │     WebSocket     │  Express + Socket.IO  │
│              │ ◄──────────────►  │                      │
└──────────────┘                    │  ┌────────────────┐  │
                                    │  │   核心引擎      │  │
                                    │  │ 事件总线        │  │
                                    │  │ 状态机          │  │
                                    │  │ 服务注册表      │  │
                                    │  └────────────────┘  │
                                    │  ┌────────────────┐  │
                                    │  │ 集成系统 (13个) │  │
                                    │  │ Demo, MQTT,     │  │
                                    │  │ Camera, Weather │  │
                                    │  │ Lutron, ISY994  │  │
                                    │  │ Yamaha, 等      │  │
                                    │  └────────────────┘  │
                                    │  ┌────────────────┐  │
                                    │  │ 自动化引擎      │  │
                                    │  │ 触发器/条件/动作│  │
                                    │  └────────────────┘  │
                                    │  ┌────────────────┐  │
                                    │  │ 注册表          │  │
                                    │  │ 实体/设备/区域  │  │
                                    │  └────────────────┘  │
                                    │  ┌────────────────┐  │
                                    │  │ better-sqlite3  │  │
                                    │  │ ha.db          │  │
```

定义项目元数据和脚本。这是一个 NestJS 项目，主要依赖:

- **@nestjs/\*** — NestJS 框架全家桶 (core, common, config, jwt, passport, platform-express, platform-socket.io, swagger, typeorm, websockets)

- **yaml / xml2js** — 配置解析

## 4. 后端: 入口与模块

### `src/main.ts` — 应用入口

**功能**: 引导 NestJS 应用，是所有请求的入口点。

工作流程:
1. 使用 `NestFactory.create<NestExpressApplication>(AppModule)` 创建应用
2. 配置静态资源服务:
   - `public/` — 前端 SPA 静态文件
   - `data/` — 用户上传文件 (GLB 模型等)
3. 配置 CORS、全局前缀 (`/api`)、全局管道 (ValidationPipe)、全局异常过滤器 (HaExceptionFilter)、全局日志拦截器
4. 设置 Swagger 文档 (`/api/doc`)
5. SPA 回退: 非 `/api/`、`/data/`、`/assets/` 路径的请求返回 `index.html`
6. 监听端口 8123

### `src/app.module.ts` — 根模块

**功能**: 导入所有功能模块，定义应用启动生命周期。

```typescript
@Module({
  imports: [
    ConfigModule.forRoot({ load: [loadHaConfig] }),  // YAML 配置
    TypeOrmModule.forRootAsync({ useFactory: getTypeOrmConfig }),  // 数据库
    CoreModule,        // 核心引擎 (全局模块)
    RegistryModule,    // 注册表
    AuthModule,        // 认证
    WebSocketModule,   // WebSocket 网关
    ApiModule,         // REST API
    IntegrationsModule, // 集成
  ]
})
```

启动时触发事件:
1. `homeassistant_start` — 开始启动
2. 加载集成
3. `homeassistant_started` — 启动完成

## 5. 后端: 核心引擎 (core)

### 5.1 `src/core/core.module.ts` — 核心模块

`@Global()` 模块，导出三个核心服务。全局模块意味着所有其他模块都可以直接注入这些服务而不需要再次导入。

### 5.2 事件总线 (`src/core/event-bus/`)

**文件**: `event-bus.service.ts`

**功能**: 基于 EventEmitter2 的发布/订阅系统，是整个系统的事件通信骨干。

```
其他服务 ──fire(event, data)──►  事件总线  ──emit──► 订阅者
                                  │
                            EventEmitter2
                          (通配符支持)
```

**核心方法**:
- `fire(eventType, data, context?)` — 触发事件
- `listen(eventType, handler)` — 订阅事件 (支持 `*` 通配符)
- `listenOnce(eventType, handler)` — 一次性订阅
- `listenMany(eventTypes, handler)` — 订阅多个事件
- `listenerCount(eventType)` — 查询订阅者数量
- `removeAllListeners(eventType?)` — 移除订阅

**预定义事件类型** (`events.constants.ts`):
- `STATE_CHANGED` — 状态变化
- `CALL_SERVICE` — 服务调用
- `SERVICE_REGISTERED / SERVICE_REMOVED` — 服务注册/移除
- `ENTITY/DEVICE/AREA_REGISTRY_UPDATED` — 注册表更新
- `HOMEASSISTANT_START / HOMEASSISTANT_STARTED / HOMEASSISTANT_STOP` — 生命周期
- `COMPONENT_LOADED` — 组件加载
- `USER_UPDATED` — 用户更新

**接口** (`ha-event.interface.ts`):
- `HaEvent<T>` — 通用事件接口
- `StateChangedData` — 状态变化事件数据
- `CallServiceData` — 服务调用事件数据
- `ComponentLoadedData` — 组件加载事件数据

### 5.3 状态机 (`src/core/state-machine/`)

**文件**: `state-machine.service.ts`

**功能**: 全局状态管理，维护所有实体的内存状态。是系统的"实时数据库"。

```
                                ┌─────────────┐
  REST API ──setState()──────►  │             │
  WebSocket ──getState()─────►  │   状态机     │──► subscribers
  集成 ──────setState()──────►  │  (Map对象)   │
                                │             │──► states_history DB
                                └─────────────┘
```

**核心接口** (`ha-state.interface.ts`):
```typescript
interface HaState {
  entity_id: string       // 如 "light.living_room"
  state: string           // 如 "on", "off", "23.5"
  attributes: Record      // 如 { brightness, friendly_name }
  last_changed: string    // 上次状态变化时间戳
  last_updated: string    // 上次更新(含属性)时间戳
  context: { id, parent_id, user_id }  // 上下文
}
```

**核心方法**:
- `setState(entityId, state, attributes?, context?)` — 设置状态 (不可变)
- `getState(entityId)` — 获取单个实体状态
- `getStates()` — 获取所有实体状态
- `hasEntity(entityId)` — 检查实体是否存在
- `removeEntity(entityId)` — 移除实体
- `getHistory(entityId, start, end?)` — 查询历史记录
- `getHistoryMultiple(entityIds, start, end?)` — 批量查询历史

**特点**:
- 状态不可变 (`Object.freeze`)
- `last_changed` 只在状态实际变动时更新; `last_updated` 每次属性变化都更新
- 启动时从数据库恢复状态
- 状态变更异步持久化到 `states_history` 表

**数据库实体** (`state.entity.ts`):
```typescript
@Entity('states_history')
class StateHistoryEntity {
  @PrimaryGeneratedColumn()  id: number
  @Column()                  entity_id: string
  @Column()                  state: string
  @Column('simple-json')     attributes: Record<string, unknown>
  @Column()                  last_changed: Date
  @Column({ nullable: true }) context_id: string
  @Column({ nullable: true }) context_user_id: string
  @Column({ nullable: true }) context_parent_id: string
}
```

### 5.4 服务注册表 (`src/core/service-registry/`)

**文件**: `service-registry.service.ts`

**功能**: 服务调用中心，将 `domain.service_name` 映射到对应的处理函数。

```
┌──────────────┐
│  REST API    │──── call("light.turn_on") ──►  服务注册表
│  WebSocket   │──── call("switch.turn_off")──►  │
│  自动化引擎  │──── call("automation.trigger")─►│  ┌────────────────┐
└──────────────┘                                  │──►│ light handler  │
                                                  │──►│ switch handler │
                                                  │──►│ automation     │
                                                  └──┴────────────────┘
```

**核心接口** (`ha-service.interface.ts`):
- `ServiceField` — 服务参数字段
- `ServiceTarget` — 服务目标 (entity_id, area_id, device_id)
- `ServiceCall` — 服务调用对象
- `ServiceHandler` — 服务处理函数类型 `(call: ServiceCall) => Promise<void>`
- `ServiceDescriptor` — 完整服务描述

**核心方法**:
- `register(domain, service, handler, descriptor?)` — 注册服务
- `call(domain, service, data, target?)` — 调用服务
- `has(domain, service)` — 检查服务是否存在
- `getDescriptor(domain, service)` — 获取服务描述
- `getDomainServices(domain)` — 获取域下的所有服务
- `getAllServices()` — 获取所有注册的服务
- `remove(domain, service)` — 移除服务
- `validate(domain, service, data)` — 验证服务参数

### 5.5 上下文 (`src/core/context/`)

**文件**: `context.service.ts`, `ha-context.interface.ts`

**功能**: 为每个操作创建跟踪上下文 (包含 ID、父 ID、用户 ID)。用于审计和追踪操作链。

## 6. 后端: 认证系统 (auth)

### `src/auth/auth.module.ts`

导入 PassportModule、JwtModule、TypeOrmModule。提供三个策略和两个守卫。

### 6.1 `auth.service.ts` — 认证服务

**功能**: 用户验证、JWT 签发、Token 管理。

**核心方法**:
- `validateUser(username, password)` — 验证用户名密码 (bcrypt 比对)
- `login(user)` — 签发 JWT access_token
- `validateToken(token)` — 验证 JWT + 长寿命 Token 回退
- `createUser(username, password, displayName)` — 创建用户
- `createLongLivedToken(userId, name)` — 创建长寿命 Token (SHA-256 哈希存储)
- `listLongLivedTokens(userId)` — 列出用户的所有 Token
- `revokeLongLivedToken(tokenId)` — 撤销 Token
- `findUserById(id)` — 查找用户
- `decodeToken(token)` — 解码 JWT

**启动行为**:
首次启动时自动创建默认管理员用户 (admin/admin)

### 6.2 `strategies/jwt.strategy.ts` — JWT 策略

从 `Authorization: Bearer <token>` 头中提取 JWT，验证签名后注入用户信息。

### 6.3 `strategies/local.strategy.ts` — 本地策略

验证用户名/密码登录。

### 6.4 `guards/jwt-auth.guard.ts` — JWT 守卫

**自定义守卫**: 首先尝试 JWT 验证，如果失败则回退到长寿命 Token (LLT) 查找。LLT 通过 `X-HA-access-token` 头或 `Authorization: Bearer` 传递。

### 6.5 `guards/local-auth.guard.ts` — 本地守卫

封装 Passport 本地策略，用于登录端点。

### 6.6 `entities/` — 数据库实体

**`user.entity.ts`**:
```typescript
@Entity('users')
class UserEntity {
  @PrimaryGeneratedColumn('uuid') id: string

  @Column({ nullable: true })     display_name: string
  @Column({ default: true })      is_admin: boolean
  @Column({ default: true })      is_active: boolean
  // 时间戳
}
```

**`long-lived-token.entity.ts`**:
```typescript
@Entity('long_lived_tokens')
class LongLivedTokenEntity {

```

### 6.7 `decorators/current-user.decorator.ts`

`@CurrentUser()` — 参数装饰器，从请求中提取当前用户。

### 6.8 `auth.controller.ts` — 认证 API

| 端点 | 方法 | 认证 | 功能 |
|------|------|------|------|
| `/api/auth/login` | POST | 无 | 用户名密码登录，返回 JWT |

## 11. 后端: 自动化引擎 (automations)

### `automation.module.ts`

整合四个核心组件: 引擎、触发器、条件、动作执行器。

### 11.1 `automation-engine.service.ts` — 自动化引擎

**核心功能**: 加载自动化配置，管理触发器绑定，处理自动化执行。

**自动化生命周期**:
```
加载配置 ──► 解析触发器 ──► 绑定事件监听 ──► 等待触发
                                                    │
                                        条件满足 ──► 执行动作序列
```

**自动化配置格式**:
```yaml
- id: "motion_light"
  alias: "Turn on living room light when motion detected"
  trigger:
    - platform: state
      entity_id: binary_sensor.motion
      to: "on"
  condition:
    - condition: state
      entity_id: light.living_room
      state: "off"
  action:
    - service: light.turn_on
      target:
        entity_id: light.living_room
  mode: single
```

**支持的执行模式**:
- `single` — 禁止并行，触发时取消之前的运行
- `parallel` — 允许多个同时运行
- `restart` — 重启模式 (默认)

**安全机制**:
- 循环保护 (防止自动化 A 触发自动化 B 又触发 A)
- 最大运行次数限制

### 11.2 `trigger-evaluator.service.ts` — 触发器求值

**支持的触发器类型**:

| 类型 | 参数 | 说明 |
|------|------|------|
| `state` | entity_id, from, to, for? | 状态变化触发，支持持续时间 |
| `numeric_state` | entity_id, above?, below?, for? | 数值阈值穿越，支持边沿检测 (>= / <=) |
| `time` | hours, minutes, seconds | 每天固定时间触发 (HH:MM) |
| `event` | event_type, event_data? | 自定义事件触发 |

**`for:` 持续时间**:
- 延迟触发，状态必须持续指定时间才触发动作

### 11.3 `condition-evaluator.service.ts` — 条件求值

**支持的条件类型**:

## 14. 前端: 上下文与状态管理 (context)

### 14.1 `HaContext.tsx` — HA 状态上下文

**功能**: React Context 提供全局 HA 状态管理。是整个前端的状态中枢。

**提供的数据**:
```typescript
interface HaCtx {
  token: string | null           // JWT Token
  login(username, password)       // 登录
  logout()                        // 登出
  wsConnected: boolean           // WebSocket 连接状态
  health: HaHealth | null        // 服务器健康状态
  states: Map<string, HaState>   // 所有实体状态
  callService(domain, service, data?, entityId?)  // 调用服务
  setEntityState(entityId, state, attributes?)    // 设置状态
}
```

**工作流程**:
1. **初始化**: 从 `localStorage` 恢复 Token
2. **登录**: POST `/api/auth/login` → 保存 Token → 设置 WebSocket
3. **WebSocket 连接**:
   - 使用 Socket.IO 连接 `/api/websocket`
   - 两阶段认证: `auth_required` → `auth` → `auth_ok`
   - 连接后请求 `get_states` + 订阅 `subscribe_events`
   - 收到 `state_changed` 事件实时更新状态 Map
4. **状态管理**: `Map<string, HaState>` — 实体 ID → 状态对象的映射
5. **登出**: 清除 Token 和状态，断开 WebSocket

### 14.2 `ToastContext.tsx` — 通知上下文

**功能**: 轻量级 toast 通知系统。

## 附录: 数据流示例

### 用户点击 3D 场景中的灯泡 → 关灯

```
1. 前端 FloorPlanPage
   ├── useSceneClick 检测点击
   ├── 提取 entityId = "light.living_room"
   ├── 调用 haSetState("light.living_room", "off")
   └── POST /api/states/light.living_room { state: "off" }

2. 后端 StatesController
   ├── JwtAuthGuard 验证 Token
   ├── 调用 stateMachine.setState("light.living_room", "off")

3. 状态机
   ├── 更新内存中的状态 Map
   ├── 计算 last_changed (状态实际变化时)
   ├── 异步写入 states_history 表
   └── 触发 STATE_CHANGED 事件

4. 事件总线
   ├── 通知 EventBusService 所有订阅者
   ├── WebSocket 网关收到事件
   └── 通过 WebSocket 推送给所有连接的前端

5. 前端 HaContext
   ├── 收到 state_changed 事件
   ├── 更新全局 states Map
   └── 触发 React 重新渲染

6. DashboardPage / FloorPlanPage
   ├── LightCard 显示 "Off"
   └── 3D 场景中灯泡熄灭, 发光消失
```

### 自动化触发流程

```
1. 集成传感器更新状态
   ├── sensor.integration 定时器触发
   ├── stateMachine.setState("sensor.temperature", "36.5")
   └── STATE_CHANGED 事件触发

2. 自动化引擎
   ├── TriggerEvaluator 检测 state 触发器
   ├── numeric_state: above 35°C → 条件满足
   ├── ConditionEvaluator 检查条件
   ├── ActionExecutor 执行动作序列
   └── callService("switch.turn_on", { entity_id: "switch.fan" })

3. 服务注册表
   ├── 调用 switch.integration 的 handler
   ├── stateMachine.setState("switch.fan", "on")
   ├── STATE_CHANGED 事件触发
   └── WebSocket 推送到前端

4. 前端收到更新
   ├── DashboardPage 显示风扇"On"
   └── (如果有 3D 映射) FloorPlanPage 更新风扇状态
```
